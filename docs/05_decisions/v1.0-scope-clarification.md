# v1.0 スコープ明確化の決定

**決定日**: 2025-02-22
**決定者**: AI Agent
**コンテキスト**: ユーザーフローと実装の齟齬分析（gap-analysis.md）に基づく

## 背景

gap-analysis.md で特定された軽微な齟齬（データエントリー画面、RUN_LINTER、設定画面）について、v1.0 MVP に含めるべきかを判断する必要があった。

## 論点

「データエントリー、RUN_LINTER、設定画面を v1.0 に含めるべきか？」

## 論点分解

### 1. データエントリー画面を v1.0 に含めるべきか？

**判断**: ❌ v1.0 には含めない（v1.1 に延期）

**理由**:
- v1.0 のスコープ決定では「レポートインポート」は含まないと明記されている
- v1.0 の目的は「モックデータで価値を検証する」こと
- データエントリー画面は外部連携（Google Slides, Notion）を前提としており、技術リスクが高い
- CEO 体験の価値検証を優先すべき

**トレードオフ**:
- メリット: 技術リスク回避、CEO 体験に集中
- デメリット: ユーザーが自分のデータで試せない

**結論**: v1.0 では Sidebar から data-entry を非表示にし、v1.1 で実装する

### 2. RUN_LINTER ボタンを v1.0 に含めるべきか？

**判断**: △ 開発モードでのみ表示（環境変数で制御）

**理由**:
- v1.0 では曖昧性検知は「完了している（モック or 自動検知済み）」という前提
- CEO の日常フローでは「再検知」は不要（Zero Latency の原則に反する）
- ただし、デバッグやデモ目的では有用

**トレードオフ**:
- メリット: ユーザーフローがシンプル、Zero Latency 原則に忠実
- デメリット: 開発時やデモ時に不便

**結論**: `NEXT_PUBLIC_ENABLE_LINTER_BUTTON=true` の環境変数でのみ表示する

### 3. 設定画面を v1.0 に含めるべきか？

**判断**: ❌ v1.0 には含めない（v1.1 に延期）

**理由**:
- v1.0 では「モックデータで検証」とされており、設定変更は不要
- CEO の日常フローに設定画面は登場しない
- KPI 定義やオーナー管理は初期セットアップであり、v1.0 の価値検証には不要

**トレードオフ**:
- メリット: CEO 体験に集中、UI がシンプル
- デメリット: カスタマイズ性が低い

**結論**: v1.0 では Sidebar から settings を非表示にし、v1.1 で実装する

## 実装方針

### Sidebar の更新

v1.1 機能を非表示にする：
- `data-entry` ビューを削除
- `settings` ビューを削除
- v1.0 では `dashboard` と `editor` のみ表示

### RUN_LINTER の制御

環境変数による表示制御：
```tsx
const showLinterButton = process.env.NEXT_PUBLIC_ENABLE_LINTER_BUTTON === 'true';
```

デフォルトでは非表示、開発時のみ `.env.local` で有効化

## 成功基準

- v1.0 では CEO の日常フロー（ダッシュボード → エディタ → 指摘対応）のみに集中
- 不要な UI 要素を排除し、「Observability」「Zero Latency」の原則を体現
- v1.1 でのデータインポート・設定管理の実装準備を整える
